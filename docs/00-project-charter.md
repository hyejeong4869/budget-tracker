# 프로젝트 헌장 (개발 규칙)

## 1. 프로젝트 개요
- 취업 포트폴리오와 개발 역량 향상을 위해 개인 프로젝트를 진행한다.
- 기존 가계부 애플리케이션은 소비 내역 중심으로 구성되어 있어 예산 중심으로 관리하기 어렵다는 불편함을 느꼈다.
- 이러한 문제를 해결하기 위해 카테고리별 예산을 먼저 설정하고, 남은 예산을 직관적으로 확인할 수 있는 예산 중심 가계부를 개발한다.
- 본 프로젝트는 실제 서비스를 개발하는 과정을 경험하기 위해 소프트웨어 공학 프로세스를 기반으로 진행한다.

## 2. 프로젝트 목표(Goal)
- Spring Boot와 REST API를 활용한 예산 중심 가계부를 개발한다.
- 소프트웨어 공학 프로세스(분석 > 설계 > 구현 > 테스트 > 회고)를 적용하여 프로젝트를 진행한다.
- 취업 포트폴리오에 활용할 수 있는 완성도 높은 개인 프로젝트를 만든다.

## 3. 프로젝트 상세 목표(Objectives)

**기능 목표**
- 회원 관리 기능 구현(회원가입/로그인)
- 예산 관리 구현(예산 설정, 카테고리 생성/수정/삭제, 지출/수입 입력, 카테고리별 남은 예산 계산, 전체 남은 예산 계산, 월별 조회)
- REST API 설계 및 구현

**프로세스 목표**
- Scrum 방식 적용
- 요구사항 분석 및 설계 문서 작성
- GitHub Issues를 활용한 작업 관리
- 테스트 및 회고 문서 작성

## 4. 프로젝트 제약사항(Constraints)
- 개발 기간은 2주를 목표로 한다.
- 기능을 무리하게 늘리지 않고 완성도를 우선한다.
- 모든 기능은 직접 구현하고 설명할 수 있는 수준을 목표로 한다.
- 추가 기능은 핵심 기능 구현 이후 진행한다.

## 5. 개발 방식(Scrum)

### 5.1 Product Backlog 작성
- 프로젝트 시작 시 구현해야 할 기능을 Product Backlog로 관리한다.

### 5.2 Sprint Planning
- Sprint 시작 전에 이번 Sprint에서 구현할 기능을 선정한다.
- Sprint는 2~3일 단위로 진행한다.

### 5.3 Sprint 진행
- 선정된 기능을 구현하며 GitHub Issues를 통해 진행 상황을 관리한다.
- Sprint 진행 과정에서 산출물(문서, 코드, 테스트)을 지속적으로 작성 및 관리한다.

### 5.4 Sprint Review
- Sprint 종료 후 구현 결과와 Sprint 산출물을 검토한다.
- Sprint 목표 달성 여부를 점검한다.

### 5.5 Sprint Retrospective
- Sprint 종료 후 잘된 점과 개선할 점을 기록하고 다음 Sprint에 반영한다.

## 6. 개발 원칙
- 요구사항 분석을 완료한 후 설계를 진행한다.
- 설계를 완료한 후 구현을 시작한다.
- 불필요한 기능 추가보다 완성도를 우선한다.
- 구현한 모든 기능은 직접 설명할 수 있어야 한다.
- 이해하지 못한 코드는 프로젝트에 포함하지 않는다.
- 구현한 기능은 가능한 한 즉시 테스트하여 정상 동작을 확인한 후 다음 작업을 진행한다.
- 프로젝트 진행 과정에서 작성한 문서는 최신 상태를 유지한다.
- 핵심 기능 구현을 우선하며, UI는 최소한의 사용성을 만족하는 수준으로 구현한다.
- 문서와 구현 내용은 항상 일치하도록 유지한다.

## 7. Git 규칙
- 프로젝트는 GitHub를 통해 관리한다.
- 프로젝트 시작부터 종료까지 모든 개발 과정을 GitHub에 기록한다.
- 문서와 소스 코드를 함께 버전 관리한다.
- 작업 단위가 완료될 때마다 커밋을 수행한다.
- Sprint 종료 시 GitHub에 산출물을 반영한다.
- 작업 시작 전 GitHub Issue를 생성하여 작업을 관리한다.
- README와 프로젝트 문서는 구현 내용에 맞게 지속적으로 업데이트한다.
- 의미 없는 커밋(예: 123, asdf, 수정)은 하지 않는다.
- 다른 사람이 봐도 이해할 수 있는 기록을 남긴다.

## 8. 커밋 규칙
- 커밋 메시지는 Conventional Commits 형식을 사용한다.
- 커밋은 하나의 작업 단위를 기준으로 작성한다.
- 하나의 커밋에는 하나의 목적만 포함한다.
- 의미 없는 커밋 메시지는 작성하지 않는다.
- 커밋 메시지는 작업 내용을 명확하게 작성한다.
- 작업 완료 후 테스트를 수행한 뒤 커밋하는 것을 원칙으로 한다.
- 하나의 커밋은 되도록 되돌릴 수 있는 최소 단위로 작성한다.
- 커밋 메시지는 다음 형식을 따른다. `<type>: <작업 내용>`

| <type>: <작업내용> | 예시                                |
| ------------------ | ----------------------------------- |
| feat: 새로운 기능  | `feat: 회원가입 API 구현`           |
| fix: 버그 수정     | `fix: 예산 계산 오류 수정`          |
| docs: 문서 수정    | `docs: 프로젝트 헌장 작성`          |
| refactor: 리팩터링 | `refactor: BudgetService 책임 분리` |
| test: 테스트 코드  | `test: BudgetService 테스트 추가`   |
| chore: 기타 작업   | `chore: 프로젝트 초기 설정`         |

## 9. 브랜치 규칙
- `main` 브랜치는 항상 안정적인 상태를 유지한다.
- 새로운 작업은 `feature/*` 브랜치에서 진행한다.
- 브랜치 이름은 `feature/<기능명>` 형식을 사용한다.
- 기능 개발이 완료되고 테스트를 마친 후 `main` 브랜치로 병합한다.
- 하나의 브랜치는 하나의 기능 개발을 원칙으로 한다.

## 10. AI 활용 원칙
- AI는 문서 작성 보조, 문서 검토, 아이디어 검토 및 개선 방안 제안을 위한 도구로 활용한다.
- AI는 요구사항 분석 및 설계 과정에서 누락된 내용이나 개선 사항을 검토하는 용도로 활용한다.
- 코드 작성 및 코드 생성에는 AI의 도움을 받지 않는다.
- 구현 과정에서 발생한 문제는 먼저 직접 원인을 분석하고 해결을 시도한 후, 필요한 경우 AI에게 오류 원인 분석이나 해결 방향을 요청한다.
- AI의 제안은 그대로 적용하지 않고, 내용을 직접 검토한 후 필요한 부분만 반영한다.
- 프로젝트에 포함되는 모든 코드와 내용은 직접 이해하고 설명할 수 있어야 한다.

## 11. 완료 기준(Definition of Done)
- 구현한 기능이 요구사항을 충족한다.
- 기능 테스트를 완료하고 정상 동작을 확인한다.
- 코드를 직접 이해하고 설명할 수 있다.
- 관련 문서를 최신 상태로 업데이트한다.
- GitHub Issue를 완료(Closed) 처리한다.
- 작업 내용을 규칙에 맞게 커밋한다.
- main 브랜치에 병합하여 안정적으로 동작하는 것을 확인한다.
- 위 완료 기준을 모두 충족하고 Sprint 목표를 달성한 경우 해당 작업을 완료로 간주한다.
